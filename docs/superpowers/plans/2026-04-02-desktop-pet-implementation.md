# ShortClaw Desktop Pet 实现计划

**日期：** 2026-04-02
**基于规范：** docs/superpowers/specs/2026-04-02-desktop-pet-design.md
**预计工作量：** 中等（约 2 到 4 天，按当前 MVP 范围）

## 概述

在 ShortClaw 中实现一个原生单角色桌宠。桌宠以独立透明窗口显示在桌面上，聚合所有 agent 状态并映射为 `idle` / `working` / `error`，支持点击唤起主窗口和右键隐藏。

本计划只覆盖当前确认后的 MVP，不包含多角色桌宠、迷你办公室、桌宠内业务操作，也不引入 Star-Office-UI 的独立后端。

## 实施策略

采用**主进程优先、会话状态优先**的接入策略，按以下顺序推进：

1. 清理设置耦合
2. 固化主进程会话隐藏语义
3. 校正桌宠窗口与 renderer 行为
4. 收口托盘/菜单/入口
5. 启动与回归验证

这样安排的原因是：

- 当前问题的根源是“设置持久化”和“真实交互语义”互相冲突
- 先拆掉设置耦合，后面的窗口与 UI 行为才不会继续打架
- 桌宠是主进程窗口能力，必须先把主进程语义收稳，再谈 renderer 表现

## 阶段 1：清理设置耦合

### 目标

把桌宠显隐彻底从设置系统中剥离，避免持久化设置和会话态语义冲突。

### 任务清单

#### 1.1 移除设置页里的桌宠区块

**涉及文件：**

- `src/pages/Settings/index.tsx`
- `src/i18n/locales/en/settings.json`
- `src/i18n/locales/zh/settings.json`
- `src/i18n/locales/ja/settings.json`

**操作：**

- 移除设置页中的桌宠显隐开关
- 移除桌宠恢复策略相关 UI
- 清理不再使用的桌宠设置文案

**验证：**

- 设置页中不再出现桌宠相关操作项
- 页面无空白残留或布局断裂

#### 1.2 清理 renderer 设置 store 中的桌宠显隐控制

**涉及文件：**

- `src/stores/settings.ts`

**操作：**

- 移除 `desktopPetEnabled` 在 renderer 设置交互中的职责
- 移除桌宠显隐 setter、同步事件消费和对应 UI 状态分支
- 若保留字段仅用于兼容已有存量数据，则明确其不再参与运行态逻辑

**验证：**

- 设置 store 初始化不再驱动桌宠显隐
- 不再存在“设置开关变化但桌宠不响应”的链路

#### 1.3 清理主进程通过设置控制桌宠的路径

**涉及文件：**

- `electron/main/ipc-handlers.ts`
- `electron/main/index.ts`
- `electron/utils/store.ts`

**操作：**

- 移除通过 `settings:set` 控制桌宠显示/隐藏的逻辑
- 停止把“隐藏桌宠”写入用户设置
- 若 `desktopPetRestorePolicy` 仍保留在持久化模型中，明确其不再参与桌宠运行时判断

**验证：**

- `settings:set` 不再承担桌宠显隐职责
- 桌宠运行态只由主进程会话逻辑决定

## 阶段 2：固化主进程会话隐藏语义

### 目标

在主进程中建立唯一正确的桌宠生命周期规则：本次隐藏后直到应用重启前不能再打开。

### 任务清单

#### 2.1 为桌宠引入会话级隐藏状态

**涉及文件：**

- `electron/main/pet-window.ts`
- `electron/main/index.ts`

**操作：**

- 在主进程引入内存态 `petSessionHidden`
- 应用启动时默认初始化为 `false`
- 用户点击“隐藏桌宠”后置为 `true`
- 应用退出后不持久化该状态

**验证：**

- 隐藏桌宠后，本次进程中状态持续有效
- 完全退出并重启后状态自动清空

#### 2.2 收口 `PetWindowController` 的显示逻辑

**涉及文件：**

- `electron/main/pet-window.ts`

**操作：**

- 所有显示桌宠的路径统一走 `PetWindowController`
- 在真正显示前先检查 `petSessionHidden`
- 若本次会话已隐藏，则直接短路，不再创建或重新显示桌宠

**验证：**

- 隐藏后无法通过其他代码路径误拉起桌宠
- 不会出现桌宠刚隐藏又被重新显示的竞态

#### 2.3 隐藏动作只影响桌宠，不影响主应用

**涉及文件：**

- `electron/main/pet-window.ts`
- `electron/main/index.ts`

**操作：**

- 隐藏桌宠时仅隐藏或销毁桌宠窗口
- 不退出主窗口
- 不停止 Gateway
- 不改动应用生命周期状态

**验证：**

- 隐藏桌宠后，ShortClaw 主应用继续正常运行
- Gateway 连接状态不受桌宠隐藏影响

## 阶段 3：校正桌宠窗口与 renderer 行为

### 目标

确保桌宠窗口始终遵循当前语义，并保持最小可用的渲染交互。

### 任务清单

#### 3.1 启动时默认显示桌宠

**涉及文件：**

- `electron/main/index.ts`
- `electron/main/pet-window.ts`

**操作：**

- 应用完成关键启动流程后默认显示桌宠
- 桌宠恢复步骤不能影响主应用或 Gateway 启动
- 若桌宠创建失败，仅记录日志，不中断主启动链路

**验证：**

- 冷启动时桌宠默认显示
- 桌宠启动失败不会导致主应用或 Gateway 初始化中断

#### 3.2 保持点击与拖动行为稳定

**涉及文件：**

- `src/pages/Pet/index.tsx`
- `electron/main/ipc-handlers.ts`
- `electron/preload/index.ts`

**操作：**

- 保持点击桌宠可唤起主窗口
- 保持拖拽能力
- 不引入新的重新显示入口

**验证：**

- 点击行为稳定
- 拖动行为稳定
- 隐藏后不会因点击/拖动逻辑把桌宠重新拉起

#### 3.3 保持三态表现与素材接入

**涉及文件：**

- `src/pages/Pet/index.tsx`
- `public/assets/pets/...`

**操作：**

- 保持 `idle` / `working` / `error` 三态渲染
- 保持当前已确认素材接入
- 不扩大为多角色或办公室模式

**验证：**

- 三态切换正常
- 素材资源能稳定加载

## 阶段 4：收口托盘、菜单与入口

### 目标

确保所有入口都遵循“隐藏后本次不可再打开”的统一规则。

### 任务清单

#### 4.1 收口桌宠右键菜单

**涉及文件：**

- `electron/main/pet-window.ts`
- `shared/pet-copy.ts`

**操作：**

- 保留：
  - 打开 ShortClaw
  - 隐藏桌宠
  - 退出 ShortClaw
- 不保留 `Show Pet`

**验证：**

- 右键菜单语义和产品规则一致

#### 4.2 收口系统托盘菜单

**涉及文件：**

- `electron/main/tray.ts`

**操作：**

- 不提供重新显示桌宠的入口
- 若保留 `Hide Pet`，其行为只隐藏当前会话桌宠
- 托盘中的桌宠项不能绕过 `petSessionHidden`

**验证：**

- 托盘不会成为绕过规则的侧门

#### 4.3 收口主进程内部显示入口

**涉及文件：**

- `electron/main/index.ts`
- 其他可能调用 `show()` 的桌宠相关模块

**操作：**

- 检查所有调用 `petWindowController.show()` 的位置
- 确保除应用冷启动外，不再存在能重新显示桌宠的调用链

**验证：**

- 隐藏后本次运行期间，任何入口都无法再次打开桌宠

## 阶段 5：启动与回归验证

### 目标

围绕最终语义做一次完整回归，确认桌宠、主应用、Gateway 三者互不干扰。

### 任务清单

#### 5.1 启动与隐藏回归

**验证用例：**

1. 启动应用，桌宠默认显示
2. 右键隐藏桌宠
3. 应用继续运行
4. 本次运行期间无法再次显示桌宠
5. 完全退出应用并重启，桌宠再次出现

#### 5.2 主应用与 Gateway 回归

**验证用例：**

1. 桌宠显示不影响主窗口启动
2. 桌宠隐藏不影响 Gateway 连接
3. 桌宠恢复失败不影响主应用和 Gateway 启动

#### 5.3 状态表现回归

**验证用例：**

1. 任一 agent 工作时桌宠显示 `working`
2. 任一 agent 错误时桌宠优先显示 `error`
3. 无工作任务时显示 `idle`

## 风险与注意事项

### 1. 旧设置残留

- 历史代码里已经存在 `desktopPetEnabled` / `desktopPetRestorePolicy`
- 实现时要避免这些旧字段继续误参与运行态判断

### 2. 启动顺序风险

- 桌宠显示属于 UI 增强能力，不应阻塞主应用和 Gateway 启动
- 启动阶段必须把桌宠恢复逻辑与主启动链路解耦

### 3. 多入口竞态风险

- 托盘、主进程、pet window 菜单都可能成为显示入口
- 必须统一走会话态检查，避免局部修复后仍被其他路径绕过

## 结论

当前 MVP 的最短正确实现路径是：

1. 先把桌宠显隐从设置系统完全移除
2. 用主进程会话内存态统一管理“隐藏后直到重启前不可再打开”
3. 保留桌宠窗口、状态聚合和最轻量交互
4. 保证桌宠逻辑不再影响主应用和 Gateway 启动

按这条计划推进，可以把当前反复出现的语义冲突、状态同步错误和启动链路回归一起收掉。
