# ShortClaw Desktop Pet 设计文档

**日期：** 2026-04-02
**状态：** 设计阶段
**作者：** Codex

## 概述

在 ShortClaw 中提供一个原生单角色桌宠。桌宠以独立透明窗口显示在桌面上，聚合 agent 工作状态，并提供最轻量的交互入口。

本设计只覆盖当前确认后的 MVP 语义，不包含多角色桌宠、迷你办公室、桌宠内业务操作，也不引入 Star-Office-UI 的独立后端或外部运行时。

## 目标

1. ShortClaw 启动后默认显示桌宠
2. 聚合所有 agent 状态，并映射为桌宠的 `idle` / `working` / `error`
3. 支持轻交互：
   - 点击桌宠唤起 ShortClaw 主窗口
   - 右键桌宠显示极简菜单并支持隐藏
4. 一旦用户点击“隐藏桌宠”，本次应用运行期间不再允许重新打开桌宠
5. 保持实现路径最短，不引入第二套应用壳或外部服务

## 非目标

- 不做多角色桌宠
- 不做迷你办公室展开态
- 不在桌宠上直接执行 agent 操作
- 不接入 Star-Office-UI 的 Python backend
- 不在设置页中继续暴露桌宠显隐开关
- 不在第一版解决所有平台上的精细视觉一致性问题

## 已确认的产品边界

### 展示形态

- 单角色桌宠
- 不展示完整办公室背景

### 状态规则

- 只要任一 agent 为 `error`，桌宠立即显示 `error`
- 否则，只要任一 agent 处于工作中，桌宠显示 `working`
- 否则显示 `idle`

### 交互规则

- 点击桌宠：唤起并聚焦 ShortClaw 主窗口
- 右键桌宠：显示极简菜单，至少提供：
  - 打开 ShortClaw
  - 隐藏桌宠
  - 退出 ShortClaw

### 生命周期规则

- 应用启动后默认显示桌宠
- 用户点击“隐藏桌宠”后：
  - 只隐藏桌宠，不退出 ShortClaw
  - 本次应用运行期间，任何入口都不能重新打开桌宠
- 只有完全重启 ShortClaw，桌宠才再次显示

## 方案选择

### 选定方案：独立 Pet 路由 + 独立 Pet 窗口 + 会话级隐藏状态

- 主进程新增一个专用 `pet window`
- Renderer 新增一个专用 `/pet` 页面
- 状态聚合逻辑抽到独立的 `PetStateService`
- 桌宠隐藏状态只保存在主进程内存中，作为会话级状态管理
- 不再通过设置系统控制桌宠显隐

### 不采用的方案

#### 方案一：在现有 Office ���面上增加 `petMode`

不采用原因：

- 当前 `Office` 页面职责过重，包含编辑、统计、Agent 卡片、热力图等完整页面能力
- 新增 `petMode` 会让页面内出现大量条件分支，形成补丁式结构
- 后续桌宠迭代会持续污染完整 Office 页面

#### 方案二：继续把桌宠显隐挂到设置系统

不采用原因：

- “本次隐藏后直到重启前都不能再打开”属于会话态，不是用户偏好设置
- 如果继续用持久化设置控制显隐，语义会和“重启后重新显示”冲突
- 会让设置状态、窗口状态、实际产品规则持续打架

#### 方案三：直接整合 Star-Office-UI 桌宠壳

不采用原因：

- 会引入第二套窗口模型和运行时边界
- 会把“桌宠功能”升级为“产品级整合”
- 与当前 MVP 的最短路径目标不符

## 架构设计

### 模块拆分

#### 1. PetStateService

职责：

- 订阅现有 agent 状态数据源
- 把多 agent 状态聚合为单一桌宠状态
- 向 pet window 推送稳定后的 `idle | working | error`

边界：

- 不负责窗口生命周期
- 不负责 UI 渲染
- 不负责设置持久化

#### 2. PetWindowController

职责：

- 创建和销毁 pet window
- 处理显示、隐藏、位置记忆、点击唤起主窗口
- 处理右键菜单
- 管理“本次会话是否已经隐藏过桌宠”的内存状态

边界：

- 不负责状态聚合
- 不负责角色动画逻辑
- 不把隐藏状态写入用户设置

#### 3. PetRenderer

职责：

- 渲染单角色桌宠
- 根据聚合状态切换动画或表现
- 处理点击事件并上报给主进程

边界：

- 不直接读取 Gateway 或 agent 原始状态
- 不承担设置管理职责

## 数据流设计

### 状态数据流

1. ShortClaw 现有 agent 状态源产生每个 agent 的运行状态
2. `PetStateService` 读取或订阅这些状态
3. `PetStateService` 执行统一聚合规则
4. 聚合结果推送给 pet window
5. `PetRenderer` 根据聚合结果切换桌宠表现

### 聚合规则

聚合优先级固定为：

1. `error`
2. `working`
3. `idle`

建议第一版把以下状态视为“工作中”：

- `working`
- `waiting`

若状态源短暂不可用，桌宠不立即回退到 `idle`，而是保留最后一个稳定状态，避免视觉闪烁和误判。

## 窗口行为设计

### pet window 基本属性

- 无边框
- 透明背景
- 置顶
- 可拖动
- 独立于主窗口生命周期存在

### 行为规则

- 应用启动后：
  - 默认创建并显示 pet window
- 点击桌宠：
  - 唤起并聚焦主窗口
- 右键桌宠：
  - 显示上下文菜单
- 用户点击“隐藏桌宠”后：
  - 立即隐藏 pet window
  - 设置主进程内存态 `petSessionHidden = true`
  - 后续本次会话中拒绝任何重新显示行为
- 关闭主窗口：
  - 不影响桌宠继续存在，除非本次会话已经隐藏过
- 退出应用：
  - pet window 随应用一起销毁
  - 会话隐藏状态自然清空

### 拖动与点击约定

第一版应显式区分拖动与点击热区，避免拖动和唤起主窗口发生冲突。建议：

- 角色主体区域支持点击唤起
- 非主体可视区域或显式拖动热区承担拖动能力

## 设置设计

### 核心原则

- 设置页不再提供桌宠显隐相关控件
- 桌宠显隐不再属于用户偏好设置
- “隐藏桌宠”是当前会话内的临时交互状态，不做持久化

### 当前结论

- 移除设置页里的桌宠设置区块
- 不再要求设置 store 管理桌宠开关
- 不再通过 `settings:set` 控制桌宠显隐

## 资源与复用策略

第一版应复用现有 `pixel-office` 里的以下能力：

- 角色 sprite 资源
- 角色状态表现逻辑
- 能独立抽离的轻量渲染能力

第一版不应复用：

- 完整 `Office` 页面结构
- 办公室布局编辑能力
- Agent 列表、统计面板、热力图等页面级 UI

原则是：

- 复用资源和状态机
- 不复用页面级复杂上下文

## 错误处理

### 桌宠状态错误

- 若 Gateway 明确进入错误态，则桌宠立即切到 `error`
- 若 agent 状态拉取失败，则桌宠切到 `error`

### 窗口级错误

- 若 pet window 创建失败：
  - 记录主进程日志
  - 不影响主窗口和 Gateway 启动

### 会话隐藏保护

- 若本次会话已经隐藏过桌宠，则所有重新显示路径都应先被短路
- 这类短路不算错误，不应弹出异常提示

## 测试要点

### 功能验证

1. 启动应用后桌宠默认显示
2. 点击桌宠可唤起主窗口
3. 右键菜单可正常显示
4. 点击“隐藏桌宠”后桌宠消失，但应用继续运行
5. 本次运行期间，任何入口都不能重新显示桌宠
6. 完全退出并重启应用后，桌宠重新出现

### 状态验证

1. 任一 agent 工作时，桌宠显示 `working`
2. 任一 agent 错误时，桌宠优先显示 `error`
3. 无工作任务时，桌宠显示 `idle`

### 稳定性验证

1. 主窗口关闭后桌宠行为符合当前会话状态
2. pet window 创建失败不会影响主应用和 Gateway 启动
3. 隐藏后不会出现桌宠被其他路径误拉起

## 结论

当前 MVP 的最短正确路径是：

- 保留独立 `pet window` 和独立 `/pet` 页面
- 用 `PetStateService` 聚合 agent 状态
- 用主进程会话内存态控制“隐藏后直到重启前不可再打开”
- 完全移除设置页中的桌宠显隐控制

这样可以避免设置持久化语义与真实交互规则冲突，同时保证桌宠、主窗口、Gateway 三者的边界清晰。
